Agent Layer ships a library of built-in skills — structured workflows that guide agents through common development tasks. Instead of describing what you want step by step, you invoke a skill and the agent follows a tested, multi-phase process with built-in quality gates.

Skills are the difference between "write me a PR" and a repeatable lifecycle that audits changes, runs CI, waits for review comments, addresses every one, and confirms CI passes before finishing. They encode the workflow discipline that makes autonomous agents trustworthy.

If you have ever watched an agent skip tests, forget to update docs, or push broken code, skills are the fix. They do not replace your judgment — they make the agent's process explicit, auditable, and consistent.

## What skills are

A skill is a Markdown file (`SKILL.md`) that defines a complete workflow for a specific task. It lives in a named directory under `.agent-layer/skills/`:

```
.agent-layer/skills/
  plan-work/
    SKILL.md
  ship-pr/
    SKILL.md
  debug-issue/
    SKILL.md
  ...
```

Each skill has:

- **A name and description** that tell the agent when to activate it
- **Defaults** so it works without configuration
- **Phased workflow steps** that guide execution from start to finish
- **Quality gates** (human checkpoints, guardrails, verification steps)
- **A defined handoff** so you know exactly what was done

Skills are projected into each client's native skill format during `al sync`. For example, they appear as `.agent/skills/` for Claude Code and `.codex/skills/` for Codex.

## How skills work

Skills use a progressive disclosure model:

1. **Catalog time** — The agent sees only the skill name and description (~50–100 tokens each). This is the routing signal that determines which skill activates for a given request.
2. **Activation time** — When a skill triggers, the full `SKILL.md` body loads into context with the complete workflow, constraints, and quality gates.
3. **Execution time** — The agent follows the phased workflow, producing artifacts, running checks, and stopping at human checkpoints when needed.

This means skills are cheap to have available (the catalog cost is small) but rich when activated. You do not pay the context cost of every skill on every request.

### Activation

Skills activate based on their `description` field. The description tells the agent both what the skill does and when it should trigger. For example:

> *Write a scoped implementation plan and task list for a requested change. Use when the user asks for a plan, roadmap slice, execution strategy, task breakdown, current roadmap-phase slice, or pre-implementation design artifact before coding.*

When you ask "plan the next roadmap phase," the agent matches that request against skill descriptions and activates `plan-work`. You can also invoke skills explicitly by name.

### Multi-agent patterns

Most skills define recommended sub-agent roles. For example, `plan-work` recommends a Scout (gathers context), Planner (drafts the plan), Critic (reviews the draft), and Execution Gatekeeper (decides if the artifacts are ready). When sub-agents are available, the skill uses them for parallel, focused work. When they are not, the agent performs each role inline.

### Artifacts

Skills produce structured output files under `.agent-layer/tmp/` using a standard naming convention:

```
.agent-layer/tmp/<skill-name>.<run-id>.<type>.md
```

For example: `.agent-layer/tmp/debug-issue.20260305-143022-a7f.report.md`

These artifacts make the agent's work inspectable and reviewable. They persist after the skill completes so you can audit what happened.

## Using skills

### Implicit activation

Just describe what you want. The agent matches your request to the right skill:

- "Plan the authentication refactor" → `plan-work`
- "Debug why the test is flaking" → `debug-issue`
- "Ship this as a PR" → `ship-pr`
- "Review the changes I just made" → `review-scope`
- "Simplify the controller layer" → `simplify-code`

### Explicit invocation

Name the skill directly when you want a specific workflow:

- "Use the `fix-issues` skill to work through ISSUES.md"
- "Run `complete-current-phase` for Phase 3"
- "Use `boost-coverage` to get the auth module above 80%"

### Providing inputs

Skills accept flexible inputs. You can provide:

- Plain-language descriptions of what you want
- Specific file paths or directories
- Constraints like "diagnosis only" or "skip the comment wait period"
- References to roadmap phases, issues, or prior artifacts

Skills define sensible defaults for everything, so you can invoke them with nothing more than a request and let the defaults handle the rest.

## The built-in skill library

Agent Layer ships 22 built-in skills organized around common development workflows. Every skill is available after `al init` — no additional setup required.

### Planning and execution

| Skill | What it does |
| --- | --- |
| `plan-work` | Writes a scoped implementation plan and task list for a requested change. Produces two artifacts: a narrative plan and an ordered task list clear enough for a fresh agent to execute. |
| `implement-plan` | Executes a plan/task-list pair without freelancing. Keeps changes aligned to the artifacts and verifies code, tests, docs, and memory before closing. |
| `complete-current-phase` | Drives a roadmap phase through planning, implementation, verification, auditing, and cleanup loops until the phase is complete. The top-level orchestrator for roadmap execution. |
| `schedule-backlog` | Proposes reviewable updates by mapping backlog items into roadmap phases, cross-checking impacts, and using human checkpoints before editing memory files. |

### Review and verification

| Skill | What it does |
| --- | --- |
| `review-scope` | Reviews files, directories, diffs, or uncommitted changes and produces a findings report covering correctness, gaps, risks, architecture, tests, docs, performance, and maintainability. |
| `review-plan` | Pre-execution review of a plan/task artifact pair. Produces findings on scope gaps, sequencing problems, and weak verification before any code is written. |
| `verify-against-plan` | Post-implementation completeness check comparing what was built against the plan. Reports gaps, regressions, missing tests or docs, and scope drift. |
| `resolve-findings` | Triages review findings, verifies each independently, implements accepted fixes, and records why findings were rejected, deferred, or resolved. |

### Quality and maintenance

| Skill | What it does |
| --- | --- |
| `simplify-code` | Assesses code complexity using metrics as diagnostic signals, removes dead code, simplifies overly complex functions, and splits files that mix unrelated responsibilities. |
| `boost-coverage` | Raises test coverage iteratively by discovering the repo's coverage commands, selecting low-coverage files, and adding tests until thresholds are met or high-value blockers emerge. |
| `repair-checks` | Runs the repo's documented checks in order, fixes failures in scope, and repeats until passing or a real blocker prevents autonomous progress. |
| `improve-codebase` | Deep autonomous codebase audit that decomposes the repo into reviewable chunks, runs parallel multi-lens reviews, fixes findings iteratively, and delegates to complementary skills. |

### Debugging and issue resolution

| Skill | What it does |
| --- | --- |
| `debug-issue` | Investigation-first debugging: reproduces the bug, narrows the cause systematically, writes a failing test, fixes the root cause, and verifies. Refuses to guess — requires evidence at every step. |
| `fix-issues` | Works through `ISSUES.md` by organizing open issues into coherent batches, planning, implementing, auditing, and keeping the issue ledger current. |

### PR and CI lifecycle

| Skill | What it does |
| --- | --- |
| `ship-pr` | Full PR lifecycle orchestrator: audits uncommitted changes, commits, pushes, creates the PR, monitors CI (fixing failures), waits for review comments, addresses every one, and confirms CI passes. |
| `fix-ci` | CI failure repair for PRs. Diagnoses failures, fixes issues, audits changes, commits, pushes, and re-checks until CI passes. |
| `address-pr-comments` | Implements fixes for agreed PR feedback, replies to every comment with justification, audits the changes, and commits/pushes. |

### Auditing

| Skill | What it does |
| --- | --- |
| `audit-and-fix-uncommitted-changes` | Working-tree audit and fix orchestrator. Runs iterative review/fix loops until convergence, with round-by-round severity reporting. Used as a pre-commit quality gate. |
| `audit-documentation` | Audits documentation for static accuracy and cross-document consistency against the repository. Optionally applies fixes for accepted findings. |
| `audit-tests` | Test suite health audit: discovers conventions, classifies tests by tier, identifies redundancy, quality gaps, and coverage gaps — without modifying tests. |
| `audit-memory` | Audits agent memory files (ISSUES, BACKLOG, ROADMAP, DECISIONS, COMMANDS, CONTEXT) for structure compliance, staleness, misplacement, and consistency. |

### Task management

| Skill | What it does |
| --- | --- |
| `finish-task` | Task closeout: checks plan alignment, updates project memory files, runs verification, and summarizes outcomes after implementation is complete. |

## Skill categories

The skills are designed to work together. Here is how they compose for common workflows:

### "I have a feature to build"

1. `plan-work` — design the approach and task list
2. `review-plan` — critique the plan before execution
3. `implement-plan` — execute the plan
4. `verify-against-plan` — confirm completeness
5. `finish-task` — close out memory and verification
6. `ship-pr` — commit, push, create PR, handle CI and reviews

### "I have a bug to fix"

1. `debug-issue` — reproduce, investigate, fix
2. `ship-pr` — get it merged

### "I want to improve code quality"

1. `improve-codebase` or `review-scope` — find issues
2. `resolve-findings` — triage and fix
3. `simplify-code` — reduce complexity
4. `boost-coverage` — fill test gaps

### "I want to drive a roadmap phase"

1. `complete-current-phase` — this orchestrator handles the entire loop internally, delegating to planning, implementation, verification, and audit skills as needed

### "I need to ship a PR"

1. `ship-pr` — handles everything from audit through merge-readiness, delegating to `audit-and-fix-uncommitted-changes`, `fix-ci`, and `address-pr-comments` automatically

## Anatomy of a skill

Every skill follows a consistent structure designed for reliable agent execution. Understanding this structure helps you read existing skills and write your own.

### Frontmatter

```yaml
---
name: debug-issue
description: >-
  Investigate a reported bug or unexpected behavior from symptom to root cause:
  reproduce, narrow down, diagnose, write a failing test, fix, and verify.
---
```

The `name` must match the directory name. The `description` is the routing signal — it determines when the skill activates. Write it for precision, not marketing.

### Defaults

Every skill defines what happens when no inputs are provided. This prevents silent guessing:

> *Default mode is full investigate-and-fix. If the user asks for diagnosis only, stop after Phase 4 and log the finding to ISSUES.md.*

### Global constraints

Hard rules that apply across all phases. These are placed early in the file because instructions appearing earlier are followed more reliably:

> *Do not attempt a fix before the bug is reproduced and the root cause is identified.*

### Human checkpoints

Specific conditions where the agent must stop and ask. These are concrete and narrow — never vague:

> *Required: ask when investigation reveals that the fix requires a breaking change, broad refactor, or architectural decision.*

### Workflow phases

The core execution steps, usually with named roles:

- **Phase 0: Preflight** — gather context, confirm baseline
- **Phase 1–N: Execution** — the actual work, broken into focused steps
- **Final phase: Verify** — run checks, confirm the work is complete

### Guardrails

Negative constraints placed at the end of the file. These catch common failure modes:

> *Do not skip reproduction and jump straight to "I think I see the problem."*

### Final handoff

What the agent reports when the skill completes — artifact paths, summary, status, and open questions.

## Writing your own skills

You can add custom skills alongside the built-in ones. Create a new directory under `.agent-layer/skills/` with a `SKILL.md` file:

```bash
mkdir -p .agent-layer/skills/my-workflow
touch .agent-layer/skills/my-workflow/SKILL.md
```

### Required fields

```yaml
---
name: my-workflow
description: >-
  One-sentence explanation of what this does and when it should trigger.
---
```

- `name`: 1–64 characters, lowercase alphanumeric and hyphens, must match the directory name.
- `description`: what the skill does and when to use it (max 1,024 characters).

### Recommended structure

Follow the same section order as the built-in skills:

1. **Opening contract** — one sentence stating what the skill does
2. **Defaults** — what happens with no inputs
3. **Inputs** — what the skill accepts
4. **Required artifacts** — file paths for output
5. **Multi-agent pattern** — recommended sub-agent roles
6. **Global constraints** — hard rules that always apply
7. **Human checkpoints** — exact conditions for asking the user
8. **Workflow phases** — ordered execution steps
9. **Guardrails** — common failure modes to avoid
10. **Final handoff** — what to report when done

### Practical tips

- **Keep it focused.** One skill, one job. If you need branching logic for different modes, split into separate skills.
- **Keep it lean.** Aim for 150–300 lines. Every token in the file carries a reasoning cost for the agent. Remove prose that does not change behavior.
- **Put critical rules early.** Instructions appearing earlier in the prompt are followed more reliably due to the primacy effect.
- **Be explicit.** Name file paths, artifact formats, and stop conditions. Prefer "stop and report Y" over "handle appropriately."
- **Design checkpoints narrowly.** "Ask when the fix requires a breaking change" is better than "ask when uncertain."

For the complete research-backed skill authoring guide, including empirical studies on instruction-following, context length effects, and constraint composition, see [Skill Design Guide](/skill-design).

### After creating a skill

Run `al sync` to project the new skill into all enabled clients. It will appear in the generated skill directories (`.agent/skills/`, `.codex/skills/`, etc.) and be available for activation immediately.

## Design philosophy

The built-in skills reflect several deliberate design choices:

### Investigation before action

Skills like `debug-issue` refuse to guess. They require reproduction, evidence, and a failing test before any fix is attempted. This mirrors how experienced developers work: understand the problem first, then solve it.

### Autonomous by default, checkpoints for real ambiguity

Skills are designed to run autonomously through their normal workflow. Human checkpoints trigger only for genuine ambiguity — scope decisions, breaking changes, or competing approaches with real tradeoffs. Vague escalation like "ask when uncertain" is deliberately avoided because it adds cognitive load without improving outcomes.

### Artifacts over memory

Skills write structured output files rather than relying on the agent to remember intermediate results. This makes the work inspectable, auditable, and recoverable. If a skill run is interrupted, the artifacts show exactly where it stopped.

### Composition over monoliths

Skills delegate to each other. `ship-pr` delegates to `audit-and-fix-uncommitted-changes`, `fix-ci`, and `address-pr-comments`. `complete-current-phase` orchestrates planning, implementation, and verification skills. This keeps each skill focused while enabling complex multi-step workflows.

### Evidence-backed structure

The skill format — critical rules early, guardrails at the end, lean bodies — is informed by research on how language models process instructions. Instructions appearing early are followed more reliably (primacy effect). Extraneous content actively degrades reasoning. Every structural choice exists to maximize the agent's reliability, not just readability.
